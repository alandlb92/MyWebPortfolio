Project Overview

This project is a real-time multiplayer Pong game rendered entirely in the console. All networking logic was implemented using C++ and Boost libraries, without any external engine or gameplay framework.

The focus was not graphics, but networking architecture, message handling, and synchronization between clients and server.

Architecture

The application follows a client-server model:

  • Server: authoritative game state management
  • Clients: input capture and state rendering

The server is responsible for:

  • Receiving player input (arrow keys)
  • Updating paddle positions
  • Simulating ball movement
  • Broadcasting updated game state

Clients only send input commands and render the state received from the server. No game logic is executed client-side, ensuring authoritative state control.

WebSocket Communication

Communication was implemented using Boost (Boost.Asio / Boost.Beast). The system handles:

  • Connection management
  • Handshake handling
  • Asynchronous message sending
  • Asynchronous message reception
  • Manual serialization of game state

Messages are lightweight and structured to transmit:

  • Player input events (UP/DOWN)
  • Ball position
  • Paddle positions

The project required explicit management of:

  • Message buffering
  • Thread safety
  • Connection lifecycle
  • Latency considerations

Console Rendering

The game is rendered directly in the console using text-based output. Each frame redraws the board using ASCII characters.

Rendering logic maps game state coordinates to console positions, allowing visual representation of:

  • Ball movement
  • Paddle positions
  • Score updates

This simplified rendering approach allowed full focus on networking logic without graphical complexity.

Input Handling

Client-side input captures arrow key presses and translates them into structured network messages sent to the server.

The server interprets these messages and updates paddle positions accordingly.

Technical Highlights

  • Low-level WebSocket implementation in C++
  • Asynchronous networking with Boost
  • Authoritative server architecture
  • Manual state synchronization
  • Console-based real-time rendering
  • No engine or external game framework

Conclusion

This study provided hands-on experience with real-time network programming, asynchronous communication patterns, and authoritative multiplayer architecture.

By removing graphical complexity and focusing purely on networking, the project allowed a deep understanding of message flow, synchronization, and server-client responsibilities in multiplayer systems.

Project Overview

This project is a simplified Space Invaders clone built entirely in C++ using SFML. The main objective was not to recreate the game itself, but to manually implement the core systems behind a real-time interactive application, including game loop, time management, collision detection, and low-level rendering logic.

Core Architecture

The application follows a traditional real-time loop structure, explicitly separating input processing, simulation updates, and rendering stages. 

while (window.isOpen())
{
    processInput();
    update(deltaTime);
    render();
}

Delta time was calculated every frame using high resolution clock measurements. All state mutations occur strictly inside the update phase, ensuring a clean separation between simulation and presentation.

Time Management & Frame Independence

Entity movement and projectile trajectories were implemented using time-scaled vector arithmetic: 

position += velocity * deltaTime;

This guarantees frame-rate independent motion and consistent gameplay behavior across different hardware configurations.

Manual Physics & Collision System

No physics middleware was used. All movement and collision detection were implemented manually using custom logic.

  • Projectile motion via vector integration
  • Enemy group translation logic
  • Boundary clamping against screen limits
  • Axis-Aligned Bounding Box (AABB) collision detection

The AABB system was implemented through custom Bounds structures, performing overlap checks using rectangle intersection logic. Collision tests were executed only for active entities to reduce unnecessary comparisons.

Data-Oriented Sprite Representation

Instead of using external textures or sprite sheets, all visual assets were defined programmatically as fixed-size binary matrices (14x14), functioning as occupancy masks.

Each sprite frame is represented as a 2D binary grid where each cell contains a 0 or 1 value, defining whether a pixel should be rendered. This effectively simulates a CPU-driven rasterization approach. 

const int Enemy2[2][196] = {
    { ... },
    { ... }
};

During rendering, the matrix is iterated manually and visible pixels are drawn as scaled quads. This avoids traditional texture loading and exposes low-level control over sprite construction.

Rendering Pipeline

Rendering was performed explicitly during the draw phase without engine-level abstractions. Entities were drawn in a deterministic order to preserve correct visual layering.

The system operates as a CPU-driven render pipeline using SFML as a thin abstraction over the graphics API.

Memory & Object Management

Entities such as Player, EnemyBlock, Barrier, and Bullet were managed manually. The project predates modern architectural patterns, so it intentionally reflects a straightforward object-oriented structure focused on clarity and experimentation rather than scalability.

Technical Learnings

  • Low-level game loop design
  • Frame-rate independent simulation
  • Manual AABB collision detection
  • Binary sprite mask representation
  • CPU-driven raster logic
  • Explicit simulation/render separation

Conclusion

This project served as a foundational technical study in real-time application architecture. By manually implementing systems that are typically abstracted by modern engines, I developed a deeper understanding of rendering pipelines, simulation loops, collision systems, and low-level data representation.
